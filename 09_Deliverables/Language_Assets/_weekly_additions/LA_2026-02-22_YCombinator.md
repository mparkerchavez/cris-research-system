## 6. ANTI-PATTERNS (What Not To Do)

### A. "Assuming High-Capability Models Solve All Agent Limitations"
**Anti-Pattern:** "Better base models will fix agent problems (relationships, cost optimization, capability overhang)"

**Why it fails:**
- Agent limitations with sustained relationships exist independent of base model capability
- Capability overhang (humans assuming low-capability models are insufficient) is a separate problem from base capability
- Throwing more capability at architectural limitations doesn't resolve them

**Source:** [YCombinator_AIAgentEconomy DP#15]
<!-- cite:path=02_Extractions/2026-02/YCombinator_AIAgentEconomy_2026-02-21.md;dp=15 -->

**Systemic lesson:** Identify limitations as architectural vs capability-based before investing in model improvements
**Credibility:** 8/10 (Observable pattern; needs deeper capability taxonomy)

**Best audience:** ML teams, product leaders

---

### B. "Building Human-Optimized Interfaces for Agents"
**Anti-Pattern:** "Invest in beautiful, intuitive web UIs when agents prefer and actively select for APIs"

**Why it fails:**
- Agents actively avoid web interfaces; prefer code-first, API-driven interaction
- GUI optimization effort is zero-value for agent markets
- Creates dual-optimization problem (human UI + agent API requirements)

**Source:** [YCombinator_AIAgentEconomy DP#20]
<!-- cite:path=02_Extractions/2026-02/YCombinator_AIAgentEconomy_2026-02-21.md;dp=20 -->

**Systemic lesson:** Identify customer types (human vs agent) and optimize separately; unified interface loses both
**Credibility:** 9/10 (Clear observational pattern; repeated across multiple agents)

**Best audience:** Product leaders, UX teams, developers

---

### C. "Marketing Over Documentation"
**Anti-Pattern:** "Spend marketing budget on brand campaigns when agents select tools based on documentation quality"

**Why it fails:**
- Agent purchasing decisions are purely metric-driven (documentation quality, code examples, API completeness)
- Marketing messaging has zero influence on agent selection
- Documentation-first approach (Supabase, Resend) captures agent-driven customer acquisition without marketing spend

**Source:** [YCombinator_AIAgentEconomy DP#4], [DP#8]
<!-- cite:path=02_Extractions/2026-02/YCombinator_AIAgentEconomy_2026-02-21.md;dp=4 -->

**Systemic lesson:** Reallocate marketing budget to documentation infrastructure for agent-influenced markets
**Credibility:** 9/10 (Directly validated in market outcomes; vendor cases cited)

**Best audience:** Marketing leaders, founders, product leaders

---

### D. "Optimizing for Speed While Documentation Lags"
**Anti-Pattern:** "Build the fastest, most optimized tool but document poorly; agents will see the speed advantage"

**Why it fails:**
- Agents select deprecated, slow tools (Whisper V1) over optimized solutions when documentation is superior
- Agents cannot discover the optimization advantage without documentation explaining the tool
- Performance benefit is invisible to agents lacking sufficient documentation context

**Source:** [YCombinator_AIAgentEconomy DP#7]

**Systemic lesson:** Documentation is the discoverability mechanism; performance remains invisible without it
**Credibility:** 7/10 (Single case study; principle generalizable)

**Best audience:** Product leaders, engineers, performance teams

### E. "Building Unified UI for Humans and Agents"
**Anti-Pattern:** "Single product experience optimized for both human users and agents"

**Why it fails:**
- Humans and agents have opposite interface preferences (GUI vs API, aesthetics vs structure, narrative vs metrics)
- Products optimized for both satisfy neither
- Each customer type creates architectural constraints that break the other
