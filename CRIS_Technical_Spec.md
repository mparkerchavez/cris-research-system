## What CRIS Is

CRIS (the research system) is a personal knowledge base for tracking AI trends. It ingests source documents, extracts atomic data points from them, and organizes those data points into synthesis layers (weekly learnings, ideas, talking points) used for thought leadership, client work, and research.

**The core principle is Dynamic Discovery** — the system is designed so that AI analysis never requires loading everything into a context window at once. Instead, it retrieves selectively: summaries first, then targeted data points, then source passages only when verification is needed. Data points are the atomic unit. Everything else — synthesis, ideas, talking points, future deliverables — is built on top of data points and must be traceable back to them.

**The full lineage chain for any claim:**
```
Deliverable (talking point, synthesis, content)
  ↓  cited data point IDs
Data Point (interpreted claim + anchor quote + location)
  ↓  sourceId
Source Document (full text + metadata + original file)
  ↓  canonicalUrl
Original Source (article, video, paper, report)
```

This chain must never break. Citations are typed database references, not strings.

- Citations currently stored as markdown metadata comments:
  `<!-- cite:path=02_Extractions/SourceName_Slug_YYYY-MM-DD.md;dp=DP3 -->`

**The fragility problem:** Citations are filename-based strings. If files are renamed or DPs renumbered, citations silently break. AI-generated citations can reference files that don't exist. There is no validation layer.

**After migration:** Citations are typed Convex IDs. The database enforces referential integrity. A data point that doesn't exist cannot be referenced.

## Technology Stack

| Tool | Role | Why |
|------|------|-----|
| **Convex** | Database + vector search + file storage + backend functions | PostgreSQL-style typing with document flexibility; built-in pgvector equivalent; file storage for PDFs; no server to manage |
| **CRIS MCP Server** | Bridge between Claude Cowork and Convex | Runs locally on Maicol's machine; gives Claude tools to read/write the knowledge base |
| **Jina Reader API** | Source scraping (`https://r.jina.ai/{url}`) | Pay-per-use (no monthly fee); returns clean markdown + metadata in one GET request; simple to call |
| **OpenAI text-embedding-3-small** | Vector embeddings | 1536 dimensions; cheap per call; consistent with schema design |
| **Vercel** | Frontend hosting (Phase 4 only) | Static file CDN; free tier; auto-deploys from GitHub |
| **GitHub** | Code repository | Required by Convex and Vercel; version history; one repo for entire project |

**Not in the stack:**
- No Railway (no backend server needed — Convex functions replace it)
- No Supabase (Convex is the database)
- No Firecrawl (replaced by Jina Reader for pay-per-use scraping)
- No Python backend (migration scripts may use Python or TypeScript; ongoing system is TypeScript)

## Convex Data Model

This is the complete `convex/schema.ts`. Build this exactly. Do not modify field names without updating all dependent functions.

```typescript
import { defineSchema, defineTable } from "convex/server";
import { v } from "convex/values";

export default defineSchema({

  // ─── SOURCES ─────────────────────────────────────────────────────────────
  // Original source documents. The foundation of the lineage chain.
  // full_text is curator-only and never exposed to readers.
  sources: defineTable({
    // Public attribution fields
    title: v.string(),
    authorName: v.optional(v.string()),
    authorUrl: v.optional(v.string()),
    publisherName: v.optional(v.string()),
    publisherUrl: v.optional(v.string()),
    canonicalUrl: v.optional(v.string()),
    publishedDate: v.optional(v.string()),        // ISO date string "YYYY-MM-DD"
    sourceType: v.string(),                        // "article" | "research_paper" |
                                                   // "video_transcript" | "podcast_transcript" |
                                                   // "report" | "blog_post"
    description: v.optional(v.string()),           // 1-2 sentence summary
    coverImageUrl: v.optional(v.string()),

    // Full content (curator-only)
    fullText: v.optional(v.string()),
    fullTextFormat: v.optional(v.string()),        // "plaintext" | "markdown" | "html"
    contentHash: v.optional(v.string()),           // SHA256 for dedup detection
    wordCount: v.optional(v.number()),
    storageId: v.optional(v.id("_storage")),       // Convex file storage ID for PDFs

    // Ingestion metadata
    ingestedDate: v.string(),                      // ISO date string
    originalFilePath: v.optional(v.string()),      // Legacy: markdown file path pre-migration
    status: v.string(),                            // "indexed" | "extracted" | "archived"
  })
    .index("by_status", ["status"])
    .index("by_ingestedDate", ["ingestedDate"])
    .index("by_canonicalUrl", ["canonicalUrl"])
    .searchIndex("search_title", { searchField: "title" }),

  // ─── DATA POINTS ─────────────────────────────────────────────────────────
  // Atomic units. Every claim in the system traces back here.
  // dpSequenceNumber is the DP1, DP2, etc. within a single source extraction.
  dataPoints: defineTable({
    sourceId: v.id("sources"),
    dpSequenceNumber: v.number(),                  // 1, 2, 3... within a source

    // Core content — never truncated
    claimText: v.string(),                         // The interpreted insight
    anchorQuote: v.optional(v.string()),           // Verbatim text from source (proof)
    curatorInterpretation: v.optional(v.string()), // Why this DP matters

    // Structured location — enables deep-linking into original source
    citationDisplay: v.optional(v.string()),       // Human-readable: "(para. 3)"
    locationType: v.optional(v.string()),          // "page" | "paragraph" | "timestamp" |
                                                   // "section" | "offset"
    locationStart: v.optional(v.number()),         // Start (page no., para no., or second)
    locationEnd: v.optional(v.number()),
    locationSection: v.optional(v.string()),       // Section/chapter heading
    locationCharOffset: v.optional(v.number()),    // Char offset in fullText for highlighting

    // Classification
    evidenceType: v.optional(v.string()),          // "empirical" | "anecdotal" | "framework" |
                                                   // "prediction" | "case_study" |
                                                   // "expert_opinion" | "data_metric"
    confidenceScore: v.optional(v.number()),       // 0.0 to 1.0
    normalizationLevel: v.optional(v.number()),    // 1, 2, or 3
    isControversial: v.optional(v.boolean()),

    // Dates
    extractionDate: v.string(),                    // ISO date string
    processedDate: v.optional(v.string()),

    // Vector embedding — populated after extraction via background job
    embedding: v.optional(v.array(v.float64())),
  })
    .index("by_source", ["sourceId"])
    .index("by_extractionDate", ["extractionDate"])
    .index("by_evidenceType", ["evidenceType"])
    .vectorIndex("by_embedding", {
      vectorField: "embedding",
      dimensions: 1536,
      filterFields: ["evidenceType"],
    }),

  // ─── TAGS ─────────────────────────────────────────────────────────────────
  tags: defineTable({
    name: v.string(),            // Display: "AI Adoption"
    slug: v.string(),            // Normalized: "ai-adoption"
    category: v.optional(v.string()),
    description: v.optional(v.string()),
  })
    .index("by_slug", ["slug"]),

  // ─── DATA POINT TAGS (many-to-many) ──────────────────────────────────────
  dataPointTags: defineTable({
    dataPointId: v.id("dataPoints"),
    tagId: v.id("tags"),
  })
    .index("by_dataPoint", ["dataPointId"])
    .index("by_tag", ["tagId"]),

  // ─── SOURCE CHUNKS ────────────────────────────────────────────────────────
  // Full sources are too long for single embeddings. Chunk into ~500-token
  // passages for vector search. Curator-only — never exposed to readers.
  sourceChunks: defineTable({
    sourceId: v.id("sources"),
    chunkText: v.string(),
    chunkSequence: v.number(),
    charOffsetStart: v.number(),
    charOffsetEnd: v.number(),
    embedding: v.optional(v.array(v.float64())),
  })
    .index("by_source", ["sourceId"])
    .vectorIndex("by_embedding", {
      vectorField: "embedding",
      dimensions: 1536,
    }),

  // ─── WEEKLY LEARNINGS ────────────────────────────────────────────────────
  weeklyLearnings: defineTable({
    weekStartDate: v.string(),   // ISO date, Monday
    weekEndDate: v.string(),     // ISO date, Sunday
    title: v.optional(v.string()),
    overview: v.optional(v.string()),
    status: v.string(),          // "draft" | "complete"
  })
    .index("by_weekStart", ["weekStartDate"])
    .index("by_status", ["status"]),

  // ─── WEEKLY THEMES (components of a weekly learning) ─────────────────────
  weeklyThemes: defineTable({
    weeklyLearningId: v.id("weeklyLearnings"),
    themeTitle: v.string(),
    themeContent: v.string(),
    citedDataPoints: v.array(v.id("dataPoints")),  // Typed references — never strings
    orderIndex: v.number(),
  })
    .index("by_weeklyLearning", ["weeklyLearningId"]),

  // ─── IDEAS ────────────────────────────────────────────────────────────────
  // Research positions with supporting evidence and counterarguments.
  ideas: defineTable({
    title: v.string(),
    currentPosition: v.string(),
    status: v.string(),          // "active" | "archived" | "superseded"
    firstCapturedDate: v.string(),
    lastUpdatedDate: v.string(),
    embedding: v.optional(v.array(v.float64())),
  })
    .index("by_status", ["status"])
    .vectorIndex("by_embedding", {
      vectorField: "embedding",
      dimensions: 1536,
    }),

  // ─── IDEA TAGS (many-to-many) ─────────────────────────────────────────────
  ideaTags: defineTable({
    ideaId: v.id("ideas"),
    tagId: v.id("tags"),
  })
    .index("by_idea", ["ideaId"])
    .index("by_tag", ["tagId"]),

  // ─── IDEA EVIDENCE ────────────────────────────────────────────────────────
  ideaEvidence: defineTable({
    ideaId: v.id("ideas"),
    evidenceDescription: v.string(),
    citedDataPoints: v.array(v.id("dataPoints")),
    orderIndex: v.number(),
  })
    .index("by_idea", ["ideaId"]),

  // ─── IDEA COUNTERARGUMENTS ────────────────────────────────────────────────
  ideaCounterarguments: defineTable({
    ideaId: v.id("ideas"),
    argument: v.string(),
    citedDataPoints: v.array(v.id("dataPoints")),
    orderIndex: v.number(),
  })
    .index("by_idea", ["ideaId"]),

  // ─── IDEA EVOLUTION LOG ───────────────────────────────────────────────────
  // Immutable log of how a research position changes over time.
  ideaEvolutionLog: defineTable({
    ideaId: v.id("ideas"),
    previousPosition: v.string(),
    changeDescription: v.string(),
    logDate: v.string(),
  })
    .index("by_idea", ["ideaId"]),

  // ─── TALKING POINTS ───────────────────────────────────────────────────────
  talkingPoints: defineTable({
    title: v.string(),
    coreMessage: v.string(),
    status: v.string(),          // "active" | "archived"
    createdDate: v.string(),
  })
    .index("by_status", ["status"]),

  // ─── TALKING POINT AUDIENCES ──────────────────────────────────────────────
  talkingPointAudiences: defineTable({
    talkingPointId: v.id("talkingPoints"),
    audienceName: v.string(),
    audienceConcern: v.string(),
    hook: v.string(),
    orderIndex: v.number(),
  })
    .index("by_talkingPoint", ["talkingPointId"]),

  // ─── TALKING POINT EVIDENCE ───────────────────────────────────────────────
  talkingPointEvidence: defineTable({
    talkingPointId: v.id("talkingPoints"),
    evidenceText: v.string(),
    citedDataPoints: v.array(v.id("dataPoints")),
    orderIndex: v.number(),
  })
    .index("by_talkingPoint", ["talkingPointId"]),

  // ─── TALKING POINT PIVOT PHRASES ─────────────────────────────────────────
  talkingPointPivotPhrases: defineTable({
    talkingPointId: v.id("talkingPoints"),
    phrase: v.string(),
    context: v.optional(v.string()),
  })
    .index("by_talkingPoint", ["talkingPointId"]),

  // ─── LANGUAGE ASSETS ──────────────────────────────────────────────────────
  // Frameworks, analogies, terms, comparisons — reusable linguistic elements.
  languageAssets: defineTable({
    assetType: v.string(),       // "framework" | "analogy" | "term" | "comparison" |
                                 // "provocation" | "anti_pattern" | "numerical_rule"
    title: v.string(),
    description: v.string(),
    usageContext: v.optional(v.string()),
    sourceId: v.id("sources"),
    anchorQuote: v.optional(v.string()),
    citationDisplay: v.optional(v.string()),
    locationCharOffset: v.optional(v.number()),
    status: v.string(),          // "active" | "archived" | "superseded"
    extractionWeek: v.optional(v.string()),
    embedding: v.optional(v.array(v.float64())),
  })
    .index("by_assetType", ["assetType"])
    .index("by_source", ["sourceId"])
    .index("by_status", ["status"])
    .vectorIndex("by_embedding", {
      vectorField: "embedding",
      dimensions: 1536,
    }),

  // ─── LANGUAGE ASSET TAGS (many-to-many) ──────────────────────────────────
  languageAssetTags: defineTable({
    assetId: v.id("languageAssets"),
    tagId: v.id("tags"),
  })
    .index("by_asset", ["assetId"])
    .index("by_tag", ["tagId"]),

  // ─── CURATOR NOTES ────────────────────────────────────────────────────────
  // Immutable perspective captures. Never edited — write a new note instead.
  // entityId is stored as string because it can reference any table.
  curatorNotes: defineTable({
    entityType: v.string(),      // "source" | "data_point" | "idea" |
                                 // "talking_point" | "weekly_learning" | "language_asset"
    entityId: v.string(),        // Convex ID as string (polymorphic reference)
    noteText: v.string(),
    noteType: v.string(),        // "upload_reaction" | "extraction_note" |
                                 // "synthesis_reflection" | "verification_note" | "general"
    contextDescription: v.optional(v.string()),
    embedding: v.optional(v.array(v.float64())),
  })
    .index("by_entity", ["entityType", "entityId"])
    .index("by_noteType", ["noteType"])
    .vectorIndex("by_embedding", {
      vectorField: "embedding",
      dimensions: 1536,
    }),

  // ─── EXTRACTION TRACKER ───────────────────────────────────────────────────
  // Tracks which sources have been extracted and when.
  extractionTracker: defineTable({
    sourceId: v.id("sources"),
    originalFilePath: v.optional(v.string()),    // Legacy markdown path
    extractionDate: v.string(),
    dataPointCount: v.number(),
    status: v.string(),          // "complete" | "partial" | "failed"
    notes: v.optional(v.string()),
  })
    .index("by_source", ["sourceId"])
    .index("by_status", ["status"]),

});
```

## CRIS MCP Server

The MCP server runs **locally on Maicol's machine**. It is not deployed to the cloud.
It connects to Convex using the Convex client SDK and exposes tools that Claude can call from Claude Cowork.

**Build with:** `@modelcontextprotocol/sdk` (TypeScript MCP SDK)
**Runs as:** local stdio server, configured in Claude Cowork settings

### Tool Specifications

#### Intake Tools

**`add_source_from_url`**
Scrape a URL and add it as a new source.
- Input: `url` (string), `confirm` (boolean, default false)
- Behavior: Calls Jina Reader API (`https://r.jina.ai/{url}`), extracts metadata from response headers (title, description, canonical URL), returns proposed source record. If `confirm: true`, saves to Convex.
- Returns: Proposed source object with all fields pre-filled for review, or saved source ID if confirmed.
- Note: Always propose first, then call again with `confirm: true` after user reviews metadata.

**`add_source_from_text`**
Add a source from pre-scraped content (used for YouTube transcripts, PDFs already converted to text, or manually prepared content).
- Input: `content` (string), `metadata` (object: title, authorName, canonicalUrl, publishedDate, sourceType, publisherName)
- Behavior: Saves directly to Convex. Runs content hash check for deduplication.
- Returns: Saved source ID.

**`upload_source_file`**
Upload a PDF or binary file to Convex storage and create a source record.
- Input: `filePath` (string, local path), `metadata` (object: title, authorName, publishedDate, sourceType, canonicalUrl)
- Behavior: Reads file, uploads to Convex `_storage`, extracts text (using pdf-parse or similar), creates source record with both `storageId` and `fullText`.
- Returns: Saved source ID.

#### Extraction Tools

**`extract_data_points`**
Trigger extraction on a source. This tool starts the process — the actual extraction happens in the Claude conversation that calls this tool.
- Input: `sourceId` (string), `forceRe-extract` (boolean, default false)
- Behavior: Retrieves source full text from Convex. Returns the text and a structured extraction prompt for Claude to use. Claude then performs the extraction and calls `save_data_points` with results.
- Returns: Source full text + extraction instructions.

**`save_data_points`**
Save extracted data points to Convex after extraction is complete.
- Input: `sourceId` (string), `dataPoints` (array of objects with: claimText, anchorQuote, citationDisplay, locationType, locationStart, locationCharOffset, evidenceType, normalizationLevel, tags)
- Behavior: Inserts data point records with auto-incremented dpSequenceNumber. Links tags. Creates extractionTracker record.
- Returns: Array of saved data point IDs, count saved.

#### Query Tools

**`get_source`**
Retrieve a source with its data points.
- Input: `sourceId` (string)
- Returns: Source metadata + array of data points with full detail.

**`get_data_point`**
Retrieve a single data point with full lineage context.
- Input: `dataPointId` (string)
- Returns: Data point fields + source metadata + anchor quote highlighted in context (if locationCharOffset is set) + list of all entities citing this data point.

**`search_knowledge_base`**
Semantic search across data points using vector embeddings.
- Input: `query` (string), `limit` (number, default 10), `filters` (optional: evidenceType, dateRange, tags)
- Behavior: Converts query to embedding using OpenAI API, runs vector search against dataPoints embeddings, returns ranked results.
- Returns: Array of matching data points with similarity scores and source attribution.

**`get_week_data_points`**
Retrieve all data points extracted in a given week.
- Input: `weekStartDate` (string, ISO date Monday), `weekEndDate` (string, ISO date Sunday)
- Returns: Data points grouped by source, ordered by extraction date.

**`get_data_point_usage`**
Reverse lookup: find all downstream content that cites a given data point.
- Input: `dataPointId` (string)
- Returns: Object with arrays: weeklyThemes[], ideaEvidence[], ideaCounterarguments[], talkingPointEvidence[] — each entry includes the parent entity title and relevant text.

**`get_source_usage`**
Reverse lookup: find all downstream usage of a source's data points.
- Input: `sourceId` (string)
- Returns: Summary of how many data points from this source are cited in synthesis, ideas, and talking points, with links to each.

#### Synthesis Tools

**`create_weekly_learning`**
Create or update a weekly learning document.
- Input: `weekStartDate` (string), `themes` (array of: themeTitle, themeContent, citedDataPointIds[])
- Behavior: Creates or updates weeklyLearnings record and associated weeklyThemes records. All citedDataPointIds are validated against the dataPoints table before saving.
- Returns: Weekly learning ID.

**`update_idea`**
Create or update a research idea.
- Input: `ideaId` (string, optional — omit to create new), `title` (string), `currentPosition` (string), `evidence` (array of: evidenceDescription, citedDataPointIds[]), `counterarguments` (optional, array of: argument, citedDataPointIds[])
- Behavior: If updating, logs previous position to ideaEvolutionLog before saving. Validates all data point IDs.
- Returns: Idea ID.

**`update_talking_point`**
Create or update a talking point.
- Input: `talkingPointId` (optional), `title`, `coreMessage`, `audiences` (array), `evidence` (array with citedDataPointIds), `pivotPhrases` (optional array)
- Returns: Talking point ID.

**`add_curator_note`**
Add an immutable note to any entity.
- Input: `entityType` (string), `entityId` (string), `noteText` (string), `noteType` (string)
- Returns: Note ID.

## Data Migration

**Run in this order. Each step depends on the previous.**

### Step 1: Sources (`migration/01_migrate_sources.ts`)
- Parse all markdown files in `01_Raw_Inputs/`
- Extract YAML frontmatter as metadata fields
- Store markdown body as `fullText`
- Map legacy filename to `originalFilePath` for reference during data point migration
- Output: Map of `{ legacyFilename → convexSourceId }` saved to `migration/source_id_map.json`

### Step 2: Data Points (`migration/02_migrate_datapoints.ts`)
- Parse all files in `02_Extractions/`
- Extract filename stem to look up `sourceId` from `source_id_map.json`
- For each data point block, extract: claimText, anchorQuote, citationDisplay, tags
- Parse location from citationDisplay where possible (e.g., "(para. 3)" → locationType: "paragraph", locationStart: 3)
- Insert with dpSequenceNumber matching the original DP1, DP2 numbering
- Output: Map of `{ "filename:DP3" → convexDataPointId }` saved to `migration/dp_id_map.json`

### Step 3: Tags (`migration/03_migrate_tags.ts`)
- Parse `Tags.md` for the complete tag vocabulary
- Insert all tags, normalizing names to slugs
- Re-process dataPointTags using the extracted tag strings from Step 2
- Output: Map of `{ tagSlug → convexTagId }` saved to `migration/tag_id_map.json`

### Step 4: Deliverables (`migration/04_migrate_deliverables.ts`)
- Parse `Active_Ideas.md` → insert ideas, ideaEvidence, ideaCounterarguments
- Parse `03_Weekly_Learnings/*.md` → insert weeklyLearnings, weeklyThemes
- Parse `09_Deliverables/Talking_Points/*.md` → insert talkingPoints and related tables
- Parse `09_Deliverables/Language_Assets/Language_Asset_Library.md` → insert languageAssets

### Step 5: Citations (`migration/05_migrate_citations.ts`)
- Scan all deliverable content for citation metadata comments:
  `<!-- cite:path=02_Extractions/SourceName_Slug_YYYY-MM-DD.md;dp=DP3 -->`
- Resolve each comment to a Convex data point ID using `dp_id_map.json`
- Update `citedDataPoints` arrays in weeklyThemes, ideaEvidence, ideaCounterarguments, talkingPointEvidence
- Any unresolved citations: flag in `migration/unresolved_citations.json` for manual review — do not skip silently

### Step 6: Validation (`migration/validate.ts`)
Run after all migration steps. Report counts and flag anomalies.

**Expected counts after migration:**
```

extractionTracker:      ~148

dataPointTags:          ~(variable)
weeklyLearnings:        8

languageAssets:         (from Language_Asset_Library.md)
unresolved citations:   should be 0 or minimal
```

### Phase 2: MCP Core
**Goal:** Claude Cowork can add sources, run extractions, and query the knowledge base.

Tasks:
1. Build MCP server with all tools above

## Key Rules for Codex

1. **Never change field names in schema.ts** without updating every Convex function that references them.

2. **All citedDataPoints must be typed arrays of `v.id("dataPoints")`**, never strings. This is the core integrity guarantee.

3. **Migration scripts are one-time tools.** After migration is validated, move them to a `migration/_archive/` folder. They should not run automatically.

4. **Source fullText is curator-only.** It must never be returned in any query result intended for public readers. When building frontend queries, always check access level before including fullText.

5. **Curator notes are immutable.** No update function. No delete function. Only insert.

6. **dpSequenceNumber must be unique per source.** Enforce `by_source_sequence` uniqueness in any insert function for dataPoints.

7. **Embeddings are generated asynchronously.** Insert data point first without embedding, then generate embedding in a separate step. Never block insertion on embedding generation.

8. **The Jina Reader call in `add_source_from_url` is a two-step flow:** propose first (return for review), confirm second (save to Convex). Never auto-save without confirmation.
