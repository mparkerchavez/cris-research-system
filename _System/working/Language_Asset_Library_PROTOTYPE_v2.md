## ASSET CATEGORY DEFINITIONS

### **1. Frameworks & Mental Models**
Conceptual structures that organize thinking about a domain. These help audiences understand "how to think about" a problem systematically.
*Example: "Tool-Coworker Duality" - AI is neither pure tool nor pure worker*

### **2. Analogies & Metaphors**
Comparisons that make abstract concepts concrete by relating them to familiar experiences.
*Example: "Coordination Tax" - the cost of agents working together (like economic tax)*

### **3. Terms & Catchphrases**
Memorable language that names phenomena, making them easier to discuss and remember.
*Example: "Vibe Coding" - non-engineers coding through feel/iteration*

### **4. Comparison Structures**
Explicit A vs B framing that clarifies distinctions between two concepts or approaches.
*Example: "Tool-Heavy vs Knowledge-Heavy Tasks" - different coordination needs*

### **5. Provocations / Reframes**
Statements that invert conventional wisdom or challenge assumptions. These shift perspective quickly and position you as a contrarian thinker.
*Example: "The challenge is organizational, not technological"*

### **6. Anti-patterns / Warnings**
Things NOT to do—memorable mistakes to avoid. Negative framing is often more memorable than positive advice.
*Example: "Don't add agents after hitting the 45% threshold"*

### **7. Numerical Rules-of-Thumb**
Memorable numbers that become verbal shorthand. These sound authoritative and are quotable.
*Example: "The 45% Rule" - coordination threshold for multi-agent systems*

### Anti-patterns / Warnings

**1. Don't Treat AI-Generated Code as Precious**
- **The Mistake:** Carefully reviewing/preserving AI output like human-written code
- **Why It Fails:** AI code is disposable; regenerate better versions instead of manually fixing
- **What to Do Instead:** Delete and regenerate with better prompts; treat code as intermediate artifact
- **Evidence:** "95% garbage rate on first attempts, 50% on second" - iteration is expected
- **Psychological Barrier:** Developers see AI assistance as "attack on identity"

**2. Don't Allocate 100% Time to Feature Development**
- **The Mistake:** Traditional allocation (all time on features, system improvement as "tech debt cleanup")
- **Why It Fails:** Compound engineering requires continuous system investment to maintain acceleration
- **What to Do Instead:** 50-50 split: 50% features, 50% system improvement
- **Evidence:** Every's operational model for sustaining single-engineer products

**3. Don't Build Manual Review Processes for AI Output**
- **The Mistake:** Human review gates for every AI-generated commit
- **Why It Fails:** Creates bottleneck that defeats speed advantage of AI; review fatigue
- **What to Do Instead:** Build automated verification infrastructure (linters, tests, CI/CD safety nets)
- **Evidence:** Operational patterns from agent-native organizations
